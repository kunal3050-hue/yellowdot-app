## Parent permissions summary

Role `parent` (mirrored in `config/permissions.js` and
`config/permissionsBackend.js`):

```
parent: ["dashboard", "profile", "fees"]
```

- `dashboard` → `/parent-home`, `/parent-attendance`
- `profile`   → `/parent-profile`, `/parent-child/:id`
- `fees`      → `/fees`
- Home route after login: `/parent-home`
- **Excluded from V1:** CCTV, self check-in, pickup history.
- Backend enforcement: `authenticate → blockUnknown → parentOnly → loadParent`;
  child access scoped to `parents.studentIds`.

## API endpoints currently active (Parent Module)

| Method | Endpoint | Returns | Notes |
|--------|----------|---------|-------|
| GET | `/api/parent/me` | `{ parent, children[] }` | provisions `parents/{uid}` on first call |
| GET | `/api/parent/children` | `{ children[] }` | child-safe projection |
| GET | `/api/parent/child/:studentId` | `{ child }` | 404 if not linked |
| GET | `/api/parent/feed` | `{ feed[] }` | announcements + events, date-sorted |

All require a parent (or bypass) account. `FeedItem = { id, type, title, body,
image, date, tag }` where `type ∈ { announcement, activity, event }`.

## Firestore collections currently used

| Collection | Used by | Access |
|------------|---------|--------|
| `parents/{uid}` | identity, profile | **new** — written by backend; read own/admin |
| `students/{id}` | profile, child profile | read (parent: own children only) |
| `announcements` | feed (Announcement + Activity) | read |
| `holidays` | feed (Event) | read |
| `notices` | feed (Event, `type=Event`) | read |

No parent-module writes to existing collections. `parents/{uid}` is the only
new collection.
